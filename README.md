# AI 开发助手使用说明

## 技能概述

`ai-dev-assistant` 是一个专门用于辅助 AI 进行开发文档编写的技能，支持计划、开发、BUG、总结等全流程文档管理。

## 核心功能

### 1. 智能初始化 ⭐ 推荐

**命令**: `auto-init`

一键完成所有初始化工作，自动执行以下步骤：
- ✅ 初始化 docs 目录（复制模板）
- ✅ 分析项目类型和技术栈
- ✅ 生成 AGENTS.md 配置文件
- ✅ 自动扫描项目模块结构
- ✅ 根据项目框架自动生成前后端开发文档
- ✅ 更新 VitePress 侧边栏配置
- ✅ 更新进度表格

**使用场景**:
- 新项目启动时
- 需要快速建立文档体系时
- 首次调用此 skill 时

**执行结果**:
- docs 目录已创建（包含所有规范、模板和功能目录）
- AGENTS.md 已生成（包含项目分析结果）
- 自动检测到项目模块并生成对应开发文档
- VitePress 侧边栏配置已更新
- 进度表格已更新

### 2. Bug 跟踪文档创建

**命令**: `create-bug-doc`

自动创建 Bug 跟踪文档，使用标准 Bug 模板。

**使用场景**:
- 发现 Bug 需要记录时
- 需要标准化 Bug 报告时
- 追踪 Bug 修复进度时

**参数**:
- `title`: Bug 标题
- `description`: Bug 描述（可选）
- `priority`: 优先级（`critical`, `high`, `medium`, `low`，默认：medium）
- `reporter`: 报告人（可选，默认：developer）

**执行结果**:
- 自动生成 Bug ID（格式：BUG-2024001）
- 填充 Bug 模板基本信息
- 保存到 Bug 管理目录

**示例**:
```bash
# 创建 Bug 文档
ai-dev-assistant create-bug-doc --title "登录页面崩溃" --priority high --reporter 测试员
```

**生成文件**:
- `docs/06-Bug 管理/BUG-2024001-登录页面崩溃.md`

### 3. 总结复盘文档创建

**命令**: `create-summary-doc`

自动创建项目总结复盘文档，使用标准总结模板。

**使用场景**:
- 项目阶段性总结
- 版本复盘
- 项目结项总结

**参数**:
- `title`: 总结标题
- `projectName`: 项目名称（可选）
- `period`: 总结周期（可选，如：2024-01）
- `author`: 作者（可选，默认：developer）

**执行结果**:
- 自动生成文件名（包含日期）
- 填充总结模板基本信息
- 保存到总结复盘目录

**示例**:
```bash
# 创建项目总结
ai-dev-assistant create-summary-doc --title "Q1 季度总结" --projectName 电商平台 --period 2024-Q1
```

**生成文件**:
- `docs/05-总结复盘/总结 -202401-Q1 季度总结.md`

### 4. 自动更新 VitePress 配置

**命令**: `update-vitepress-config`

自动扫描 docs 目录，更新 VitePress 站点的侧边栏配置。

**使用场景**:
- 新增开发文档后
- 删除文档后
- 需要更新导航结构时

**执行结果**:
- 自动读取所有 Markdown 文件的 Frontmatter
- 提取 title 字段作为导航文本
- 更新 `.vitepress/config.ts` 中的 sidebar 配置
- 保持侧边栏与文档同步

**示例**:
```bash
# 新增文档后调用
ai-dev-assistant update-vitepress-config
```

### 5. 自动更新进度表格

**命令**: `update-index-files`

自动扫描开发文档，更新 index.md 中的进度表格。

**使用场景**:
- 创建新模块文档后
- 更新文档状态后
- 需要查看整体进度时

**执行结果**:
- 自动读取 `03-前端开发与计划/` 和 `04-后端开发与计划/` 下的所有文档
- 解析 Frontmatter 中的状态、进度、负责人等信息
- 更新 index.md 中的进度表格
- 保持进度信息实时准确

**示例**:
```bash
# 更新进度后调用
ai-dev-assistant update-index-files
```

## 文档编写规则（必须遵守）

### Frontmatter 规范

所有文档必须包含以下 Frontmatter：

```yaml
---
title: 文档标题（必填）
description: 文档描述，用于 SEO（必填）
category: 分类（文档规范/前端开发/后端开发/总结/Bug/测试/运维/变更）
tags:
  - 标签 1
  - 标签 2
outline: deep（或数字 2/3）
---
```

**建议字段**（根据文档类型选择）:
```yaml
# 开发文档
module: 模块名称
status: planning | developing | completed | on-hold
progress: 0-100
priority: low | medium | high | critical
owner: 负责人
created: YYYY-MM-DD
updated: YYYY-MM-DD

# Bug 文档
status: new | investigating | fixed | verified | closed
priority: low | medium | high | critical
bugId: BUG-001

# 测试文档
testType: manual | automated | performance | security
status: planning | in-progress | completed
```

**开发文档**:
- 格式：`三位数字 - 模块名称开发.md`

**Bug 文档**:
- 格式：`BUG-三位数字-Bug 标题.md`
- 示例：`BUG-001-登录页面验证码不显示.md`

**总结文档**:
- 格式：`总结-YYYYMMDD-主题.md`
- 示例：`总结 -20240115-V1.0 版本复盘.md`

**测试文档**:
- 格式：`TEST-三位数字 - 测试名称.md`
- 示例：`TEST-001-用户管理模块测试.md`

#### ❌ 不要写进 docs 的内容
- 代码细节（应该在代码注释中）
- 临时笔记（应该整理后写入）
- 个人随笔（与开发无关的内容）
- 敏感信息（密码、密钥等）

### 文档编写原则

1. **一个模块一个文档** - 不要过度拆分
2. **平铺管理** - 不拆子目录，便于检索
3. **按开发顺序** - 从项目构建到上线
4. **真实状态** - 记录真实开发状态，不写空泛描述
5. **与代码一致** - 文档内容要和代码现状保持一致
6. **及时更新** - 开发过程中实时更新文档
7. **图类优先** - 架构图、流程图优先放到总结复盘

## 常见问题

### Q: 为什么要用 docs 目录？
A: 统一管理开发文档，支持 VitePress 生成美观的文档站点，便于团队协作和知识沉淀。

### Q: 文档应该什么时候写？
A: 开发前写计划，开发中写进度，开发后写总结。Bug 和测试随时记录。

### Q: 如何保证文档质量？
A: 严格按照模板格式，记录真实状态，定期 review 和更新。

### Q: 文档太多找不到怎么办？
A: 使用统一的命名规范，通过文件名即可快速检索。VitePress 站点也支持搜索。
