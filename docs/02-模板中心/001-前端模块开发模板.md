# 【模块名称】前端模块开发文档

> ⚠️ **核心原则：模板服务于项目，项目优先于模板**
> 
> **当模板与项目冲突时，必须修改模板适配项目，绝不允许让项目适配模板！**
> 
> 💡 **使用说明**：
> 
> 1. **第一步：项目分析** - 运行技能分析现有项目结构
> 2. **第二步：识别差异** - 对比模板与项目的差异点
> 3. **第三步：修改模板** - 根据项目特点调整模板内容
> 4. **第四步：生成文档** - 使用适配后的模板生成文档
> 
> 🚨 **强制规则**：
> - ❌ 禁止直接套用模板
> - ❌ 禁止修改现有代码风格适配模板
> - ✅ 必须修改模板适配现有项目
> - ✅ 必须保持与现有代码的一致性

```typescript
  id: string

}

export const use【模块名】Store = defineStore('【模块名】', () => {
  // State
  const items = ref<【实体名】[]>([])
  const loading = ref(false)
  
  // Getters
  const itemCount = computed(() => items.value.length)
  
  // Actions
  async function fetchItems() {
    loading.value = true
    try {
      const response = await api.getItems()
      items.value = response.data
    } catch (error) {
      console.error(error)
    } finally {
      loading.value = false
    }
  }
  
  return {
    items,
    loading,
    itemCount,
    fetchItems,
  }
})
```

```typescript
        commit('SET_LOADING', false)
```

## 十一、二次开发最佳实践

### 11.1 理解优先

✅ **先理解现有代码，再开始开发**
- 阅读现有代码了解架构
- 理解设计决策背后的原因
- 不要急于否定现有实现

### 11.2 保持一致

✅ **与现有代码保持一致**
- 目录结构一致
- 命名规范一致
- 代码风格一致
- 不要混用多种风格

### 11.3 小步迭代

✅ **小步快跑，频繁验证**
- 不要一次性重写大量代码
- 每次改动保持可回滚
- 频繁运行测试验证

### 11.4 向后兼容

✅ **保证向后兼容**
- 不破坏现有 API
- 不删除现有功能
- 如需破坏性变更，提供迁移方案

### 11.5 代码复用

✅ **优先复用现有代码**
- 不要重复造轮子
- 扩展现有工具函数
- 继承现有组件基类

### 11.6 渐进优化

✅ **渐进式优化，不要一步到位**
- 优先完成功能开发
- 在功能稳定后逐步优化
- 优化时保证测试覆盖

## 十二、常见问题

### Q1: 现有代码风格不一致怎么办？

**A:** 
1. 优先遵循现有**主要**风格
2. 在新模块中使用一致的规范
3. 可以逐步重构现有代码
4. 不要引入新的风格

### Q2: 是否需要为现有代码补充类型？

**A:**
1. 优先保证**新代码**的类型完整
2. 在使用现有代码时逐步补充
3. 不要一次性重写所有类型
4. 避免大规模重构

### Q3: 如何处理技术债务？

**A:**
1. 记录技术债务清单
2. 评估影响和优先级
3. 在迭代中逐步解决
4. 避免引入新的技术债务

### Q4: 如何平衡开发速度和代码质量？

**A:**
1. 核心功能保证质量
2. 非核心功能可以适当妥协
3. 记录妥协点，后续优化
4. 保证测试覆盖底线
